# PRD 샘플 — RAG Orchestrator Agent

> 이 문서는 **제품 요구사항(PRD)** 작성 예시입니다. 실제 프로젝트에서는 목차·용어·승인 절차를 조직 표준에 맞게 조정하세요.

## 1. 문서 정보

| 항목 | 내용 |
|------|------|
| 제목 | RAG 기반 멀티에이전트 오케스트레이션 서비스 |
| 버전 | 0.1 (샘플) |
| 상태 | 초안 |
| 관련 이해관계자 | 기획, 백엔드, 프론트, 보안 |

## 2. 배경·문제 정의

- 사용자가 사내/프로젝트 문서를 바탕으로 **근거 있는 답변**을 받고 싶다.
- 단일 LLM 호출만으로는 최신·비공개 지식 반영이 어렵다.

## 3. 목표·비목표

### 목표

- 문서를 **간단한 DB(벡터 저장)**에 넣고, 질의 시 **검색 + 생성** 파이프라인으로 답한다.
- **3개 이상의 전문 에이전트**를 **오케스트레이터**가 순서대로 조율한다.
- 문서 입출력 경로는 **MCP**로 표준화해 에디터·다른 클라이언트에서도 동일 도구를 쓸 수 있게 한다.

### 비목표 (초기 버전)

- 멀티 테넌트·세분화된 RBAC
- 실시간 협업 편집

## 4. 사용자·시나리오

| 페르소나 | 시나리오 |
|----------|----------|
| 내부 사용자 | 정책 문서를 인제스트한 뒤, 규정 질문에 답을 받는다. |
| 개발자 | MCP로 동일 DB에 도구 호출을 붙여 자동화한다. |

## 5. 기능 요구사항

| ID | 요구사항 | 우선순위 | 비고 |
|----|----------|----------|------|
| F-01 | 텍스트 인제스트(선택적 청크 분할) | Must | MCP `rag_ingest` |
| F-02 | 유사도 검색 + 선택적 재순위 | Must | MCP `rag_search` |
| F-03 | 질의에 대한 오케스트레이션(계획→검색→합성→검증) | Must | REST `/api/chat` |
| F-04 | SSE 스트리밍 응답 | Should | `/api/chat/stream` |
| F-05 | SK CI를 참고한 웹 UI | Could | 데모 수준 |

## 6. 비기능 요구사항

- **보안**: API 키·환경변수는 저장소에 포함하지 않는다.
- **관측**: 에이전트 단계별 트레이스를 응답에 포함할 수 있어야 한다.
- **성능**: 동시 사용자 수 목표는 _TBD_ (PRD에서 수치 확정).

## 7. 가정·제약

- Chroma 로컬 경로 영속 저장.
- OpenAI 호환 API가 없을 경우 합성은 근거 텍스트 폴백.

## 8. 릴리스·검수

- **MVP**: 인제스트 → 단일 질의 → 근거·답변·트레이스 확인.
- **검수**: 샘플 문서 3건으로 회귀 시나리오 체크리스트 수행.
